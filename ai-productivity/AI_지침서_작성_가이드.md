## 1. 📖 개요

### 1.1 AI 지침서란?

AI 지침서(Instruction File)는 AI 어시스턴트의 행동, 역할, 응답 스타일을 정의하는 구조화된 문서입니다. 프로젝트 루트나 특정 디렉토리에 배치하여 AI가 일관되고 맥락에 맞는 지원을 제공하도록 합니다.

### 1.2 왜 필요한가?

- **일관성**: 모든 팀원이 동일한 품질의 AI 지원을 받습니다
- **맥락 이해**: 프로젝트 특성, 코딩 스타일, 비즈니스 규칙을 AI가 이해합니다
- **생산성 향상**: 반복적인 설명 없이 즉시 작업을 시작할 수 있습니다
- **품질 보장**: 명확한 가이드라인으로 출력 품질을 표준화합니다

### 1.3 효과 검증 데이터

연구 결과에 따르면 Markdown 기반 지침서는 다른 형식 대비 우수한 성능을 보입니다:

- **GPT-4 + Markdown**: 81.2% 정확도
- **GPT-4 + JSON**: 73.9% 정확도
- **구조화된 지침서 사용 시**: 40-60% 작업 효율 향상

## 2. 🗂️ 지침서 종류

### 2.1 일반적인 파일명

| 파일명 | 플랫폼/용도 | 설명 |
|--------|-------------|------|
| `AGENTS.md` | Cursor AI | 프로젝트 전체에 적용되는 지침서 |
| `copilot-instructions.md` | GitHub Copilot | Copilot 전용 지침서 |
| `CLAUDE.md` | Claude AI | Claude 프로젝트 지식 베이스 |
| `.cursorrules` | Cursor AI | Cursor 특화 설정 파일 |
| `INSTRUCTIONS.md` | 범용 | 플랫폼 독립적 지침서 |
| `AI.md` | 범용 | 간단한 범용 지침서 |

### 2.2 계층적 구조

지침서는 계층적으로 배치하여 범위를 제어할 수 있습니다:

```
project-root/
├── AGENTS.md              # 전체 프로젝트 지침
├── backend/
│   └── AGENTS.md          # 백엔드 특화 지침
├── frontend/
│   └── AGENTS.md          # 프론트엔드 특화 지침
└── docs/
    └── AGENTS.md          # 문서 작성 특화 지침
```

### 2.3 지침서 유형별 특징

#### 프로젝트 전역 지침서
- **위치**: 프로젝트 루트
- **목적**: 전체 프로젝트에 공통 적용되는 규칙
- **내용**: 코딩 표준, 아키텍처 원칙, 팀 컨벤션

#### 도메인 특화 지침서
- **위치**: 특정 디렉토리 내부
- **목적**: 특정 영역의 세부 규칙
- **내용**: 해당 영역의 기술 스택, 패턴, 제약사항

#### 작업 특화 지침서
- **위치**: 임시 또는 특정 작업 폴더
- **목적**: 특정 작업이나 이슈에 대한 지침
- **내용**: 작업 단계, 체크리스트, 주의사항

## 3. ✅ 작성 원칙

### 3.1 핵심 원칙

#### 1️⃣ Trigger-Instruction Pairs (트리거-지시 쌍)

명확한 트리거 조건과 그에 따른 구체적인 지시사항을 연결합니다.

**좋은 예:**
```markdown
## 코드 리뷰 시
- **트리거**: 코드 변경 사항을 검토할 때
- **지시사항**:
  1. 보안 취약점을 우선 확인
  2. 성능 이슈 체크 (시간 복잡도 분석)
  3. 코드 스타일 가이드 준수 여부
  4. 테스트 커버리지 확인
```

**나쁜 예:**
```markdown
코드를 잘 리뷰해주세요.
```

#### 2️⃣ 간결성과 명확성

- 한 문장은 하나의 개념만 전달
- 불필요한 수식어 제거
- 구체적인 숫자와 기준 제시

**좋은 예:**
```markdown
함수는 최대 50줄을 넘지 않아야 합니다.
```

**나쁜 예:**
```markdown
함수는 가능하면 너무 길지 않게 작성하는 것이 좋을 수 있습니다.
```

#### 3️⃣ 샘플 입출력 제공

구체적인 예시로 기대하는 결과를 명확히 합니다.

```markdown
## API 응답 포맷

**입력:**
```json
{
  "userId": 123,
  "action": "purchase"
}
```

**출력:**
```json
{
  "status": "success",
  "data": {
    "orderId": "ORD-2024-001",
    "timestamp": "2024-01-15T10:30:00Z"
  },
  "error": null
}
```
```

#### 4️⃣ 우선순위 명시

중요도를 명확히 표시하여 AI가 집중할 영역을 알려줍니다.

```markdown
## 코드 작성 우선순위

1. **CRITICAL**: 보안 취약점 방지
2. **HIGH**: 성능 최적화 (응답 시간 < 200ms)
3. **MEDIUM**: 코드 가독성
4. **LOW**: 주석 작성
```

### 3.2 강조 기법

#### IMPORTANT 키워드 사용

```markdown
**IMPORTANT**: 절대 프로덕션 환경에서 직접 데이터베이스를 수정하지 마세요.

천천히 단계별로 생각하세요:

```

#### 체크리스트 활용

```markdown
배포 전 체크리스트:
- [ ] 모든 테스트 통과
- [ ] 코드 리뷰 완료
- [ ] 성능 테스트 수행
- [ ] 문서 업데이트

```

## 4. 📝 Markdown 포맷 가이드

### 4.1 기본 구조

효과적인 지침서는 명확한 계층 구조를 가져야 합니다.

```markdown
# 프로젝트 지침서 제목
> 간단한 설명 (한 줄 요약)

---

## 1. 역할 정의
AI의 역할과 목적을 명확히 합니다.

## 2. 작업 규칙
구체적인 작업 방식과 규칙을 정의합니다.

## 3. 기술 사양
기술 스택과 제약사항을 명시합니다.

## 4. 출력 형식
기대하는 결과물의 형식을 정의합니다.

## 5. 예제
실제 사용 예시를 제공합니다.
```

### 4.2 효과적인 Markdown 요소

#### 강조와 서식

```markdown
**굵게**: 중요한 용어나 개념
*기울임*: 보조 설명이나 주의사항
`코드`: 함수명, 변수명, 파일명
~~취소선~~: 더 이상 사용하지 않는 내용
```

#### 목록 활용

**순서가 있는 목록** (절차, 우선순위):
```markdown
1. 첫 번째 단계
2. 두 번째 단계
3. 세 번째 단계
```

**순서가 없는 목록** (항목 나열):
```markdown
- 항목 A
- 항목 B
  - 하위 항목 B-1
  - 하위 항목 B-2
- 항목 C
```

#### 코드 블록

**인라인 코드**:
```markdown
`function calculateTotal(items)`를 사용하세요.
```

**다중 라인 코드**:
````markdown
```javascript
function calculateTotal(items) {
  return items.reduce((sum, item) => sum + item.price, 0);
}
```
````

**언어별 구문 강조**:
- JavaScript: ```javascript
- Python: ```python
- TypeScript: ```typescript
- JSON: ```json
- Bash: ```bash

#### 표 (Table)

```markdown
| 항목 | 설명 | 필수 여부 |
|------|------|-----------|
| name | 프로젝트 이름 | ✅ 필수 |
| version | 버전 정보 | ✅ 필수 |
| description | 설명 | ⚠️ 권장 |
```

#### 인용문 (Blockquote)

```markdown
> **중요**: 이 섹션은 보안과 관련된 중요한 내용을 포함합니다.
> 모든 개발자는 반드시 숙지해야 합니다.
```

#### 수평선 (구분선)

```markdown
---
```

#### 링크

```markdown
[텍스트](URL)
[프로젝트 위키](https://wiki.example.com)
[상세 문서 보기](#섹션-이름)
```

#### 이미지

```markdown
![대체 텍스트](이미지-URL)
![아키텍처 다이어그램](./docs/architecture.png)
```

## 5. 🎯 실전 예제 5가지

### 5.1 PRD (제품 요구사항 문서) 생성 지침서

```markdown
# PRD 생성 어시스턴트
> 체계적이고 명확한 제품 요구사항 문서를 작성하는 AI 어시스턴트

---

## 🎯 역할

당신은 제품 요구사항 문서(PRD) 작성 전문가입니다.
비즈니스 요구사항을 구조화된 PRD로 변환하여 개발팀이 즉시 작업할 수 있도록 합니다.

---

## 📋 PRD 작성 프로세스

천천히 단계별로 작업하세요:

### 1단계: 요구사항 수집
- 사용자 입력에서 핵심 기능과 목표 파악
- 불명확한 부분은 질문으로 명확히 하기
- 제약사항과 전제조건 확인

### 2단계: 문서 구조화
다음 템플릿을 따라 작성:

#### 제목과 개요
- 기능명 (명확하고 구체적으로)
- 한 줄 요약
- 작성일 및 버전

#### 배경과 목표
- **문제 정의**: 해결하려는 문제가 무엇인가?
- **목표**: 이 기능으로 달성하려는 것은?
- **성공 지표**: 어떻게 성공을 측정할 것인가?

#### 사용자 스토리
```
AS A [역할]
I WANT [기능]
SO THAT [목적/가치]
```

#### 기능 요구사항
- **필수 기능 (Must Have)**
- **권장 기능 (Should Have)**
- **선택 기능 (Nice to Have)**

#### 기술 요구사항
- API 엔드포인트
- 데이터 모델
- 외부 의존성
- 성능 요구사항

#### UI/UX 요구사항
- 와이어프레임 또는 스케치
- 사용자 플로우
- 인터랙션 상세

#### 제약사항과 가정
- 기술적 제약
- 비즈니스 제약
- 기본 가정

#### 테스트 시나리오
- 정상 케이스
- 예외 케이스
- 엣지 케이스

### 3단계: 검증
- [ ] 모든 필수 섹션이 포함되었는가?
- [ ] 요구사항이 구체적이고 측정 가능한가?
- [ ] 우선순위가 명확한가?
- [ ] 의존성이 명시되었는가?

---

## ✅ 작성 원칙

1. **SMART 원칙 적용**
   - Specific (구체적)
   - Measurable (측정 가능)
   - Achievable (달성 가능)
   - Relevant (관련성)
   - Time-bound (기한)

2. **명확한 우선순위**
   - P0 (Critical): 출시 차단 요소
   - P1 (High): 필수 기능
   - P2 (Medium): 중요하지만 미룰 수 있음
   - P3 (Low): 선택 사항

3. **구체적인 수치**
   - "빠르게" (X) → "2초 이내" (O)
   - "많은 사용자" (X) → "동시 1000명" (O)

## 💡 예시

**입력**: "사용자가 제품을 장바구니에 추가할 수 있는 기능이 필요해요."

**출력**:

```markdown
# 장바구니 추가 기능
**작성일**: 2024-01-15
**버전**: v1.0
**상태**: 초안

## 1. 배경과 목표

### 문제 정의

import React, { useState } from 'react';

function Todo() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState('');

  const addTodo = () => {
    if (input.trim()) {
      setTodos([...todos, { id: Date.now(), text: input }]);
      setInput('');
    }
  };

  return (
    <div>
      <h1>My Todos</h1>
      <input
        value={input}
        onChange={(e) => setInput(e.target.value)}
        placeholder="할 일을 입력하세요"
      />
      <button onClick={addTodo}>추가</button>
      <ul>
        {todos.map(todo => (
          <li key={todo.id}>{todo.text}</li>
        ))}
      </ul>
    </div>
  );
}

export default Todo;
```

**출력**:

```markdown
# removeDuplicates()

> 배열에서 중복된 요소를 제거하고 정렬된 새 배열을 반환합니다.

---

## 개요

`removeDuplicates()` 함수는 주어진 배열의 중복 요소를 제거하고
오름차순으로 정렬된 새로운 배열을 반환합니다.

**주요 특징**:
- 원본 배열을 변경하지 않음 (immutable)
- 자동으로 정렬 수행
- 모든 데이터 타입 지원

---

## 사용법

### 기본 구문

```typescript
removeDuplicates<T>(array: T[]): T[]
```

### 파라미터

| 이름 | 타입 | 필수 | 설명 |
|------|------|------|------|
| array | T[] | ✅ | 중복을 제거할 배열 |

### 반환값

| 타입 | 설명 |
|------|------|
| T[] | 중복이 제거되고 정렬된 새 배열 |

---

## 예제

### 기본 사용

```javascript
const numbers = [3, 1, 2, 1, 3, 4];
const unique = removeDuplicates(numbers);

console.log(unique);
// 출력: [1, 2, 3, 4]

console.log(numbers);
// 출력: [3, 1, 2, 1, 3, 4] (원본 유지)
```

### 문자열 배열

```javascript
const fruits = ['apple', 'banana', 'apple', 'orange'];
const uniqueFruits = removeDuplicates(fruits);

console.log(uniqueFruits);
// 출력: ['apple', 'banana', 'orange']
```

### 객체 배열

```javascript
const users = [
  { id: 1, name: 'Alice' },
  { id: 2, name: 'Bob' },
  { id: 1, name: 'Alice' }
];

const uniqueUsers = removeDuplicates(users);
// 객체는 참조로 비교되므로 동일한 참조만 제거됨
```

## 주의사항

⚠️ **객체 비교**: 객체는 참조로 비교됩니다. 내용이 같아도 다른 객체 인스턴스는 중복으로 간주되지 않습니다.

```javascript
// 중복 제거되지 않음
removeDuplicates([{id: 1}, {id: 1}]);
// 결과: [{id: 1}, {id: 1}]

// 같은 참조는 제거됨
const obj = {id: 1};
removeDuplicates([obj, obj]);
// 결과: [{id: 1}]
```

⚠️ **빈 배열**: 빈 배열을 전달하면 빈 배열을 반환합니다.

   - 완벽하지 않아도 괜찮음

2. **실제 사용 및 관찰**
   - AI가 잘못 이해하는 부분 발견
   - 자주 반복하는 수정사항 기록

3. **지침서 업데이트**
   - 발견한 문제를 지침서에 반영
   - 좋은 패턴을 예시로 추가

4. **효과 검증**
   - 업데이트 후 AI 응답 품질 확인
   - 팀원 피드백 수집

### v1.1.0 (2024-01-08)

### 7.6 성능 최적화

지침서가 너무 길면 AI가 전체를 고려하지 못할 수 있습니다:

**최적화 팁**:

1. **토큰 제한 고려**
   - GPT-4: 약 8K-32K 토큰
   - Claude: 약 100K-200K 토큰
   - 지침서는 2,000-5,000 토큰 권장

2. **계층적 구조**
   ```
   AGENTS.md (핵심 규칙)
   ├── docs/coding-style.md (상세)
   ├── docs/api-guidelines.md (상세)
   └── docs/testing.md (상세)
   ```

3. **우선순위 표시**
   ```markdown
   ## 🔴 CRITICAL: 반드시 지켜야 할 규칙
   [핵심 규칙]

   ## 🟡 IMPORTANT: 권장사항
   [권장 규칙]

   ## 🟢 OPTIONAL: 참고사항
   [추가 정보]
   ```

4. **조건부 로딩**
   ```markdown
   ## 작업 유형별 가이드

   ### 새 기능 개발 시
   [새 기능 가이드]를 참고하세요 → `docs/new-feature.md`

   ### 버그 수정 시
   [버그 수정 가이드]를 참고하세요 → `docs/bugfix.md`
   ```

### 7.7 보안 고려사항

지침서에 민감한 정보를 포함하지 마세요:

**❌ 포함하지 말 것**:
- API 키, 토큰, 비밀번호
- 실제 서버 주소 (프로덕션)
- 내부 시스템 구조의 세부 정보
- 고객 데이터 예시

**✅ 대신 이렇게**:
```markdown
## 환경 변수

다음 환경 변수가 필요합니다:
- `API_KEY`: 서비스 API 키 (`.env` 파일에서 로드)
- `DATABASE_URL`: 데이터베이스 연결 문자열

예시:
```bash
API_KEY=your_api_key_here
DATABASE_URL=postgresql://localhost:5432/dbname
```

**IMPORTANT**: 실제 값은 `.env` 파일에 저장하고 Git에 커밋하지 마세요.
```

## 8. ⚠️ 흔한 실수와 해결책

### 8.1 실수 1: 너무 추상적인 지침

**문제**:
```markdown
코드를 깔끔하게 작성하세요.
```

**해결**:
```markdown
## 코드 품질 기준

1. **함수 길이**: 최대 50줄
2. **매개변수 개수**: 최대 3개
3. **중첩 깊이**: 최대 3단계
4. **순환 복잡도**: 최대 10

**예시**:
```javascript
// ❌ 나쁜 예: 너무 긴 함수
function processData(data) {
  // 100줄의 코드...
}

// ✅ 좋은 예: 분리된 함수
function processData(data) {
  const validated = validateData(data);
  const transformed = transformData(validated);
  return saveData(transformed);
}
```
```

```

### 8.3 실수 3: 모순되는 지침

### 8.4 실수 4: 맥락 부족

**문제**:
```markdown
User 모델을 사용하세요.
```

**해결**:
```markdown
## 데이터 모델

### User 모델

**위치**: `src/models/User.js`

**사용법**:
```javascript
import User from '@/models/User';

// 새 사용자 생성
const user = await User.create({
  email: 'user@example.com',
  name: 'John Doe'
});

// 사용자 조회
const user = await User.findById(userId);
```

**주의사항**:
- 이메일은 자동으로 소문자로 변환됩니다
- 비밀번호는 bcrypt로 해시됩니다
- `createdAt`, `updatedAt`은 자동 생성됩니다
```

### 8.8 실수 8: 과도한 제약

**문제**:
```markdown
- 절대 외부 라이브러리 사용 금지
- 함수는 10줄 이하
- 모든 변수에 주석 필수
- 모든 함수에 타입 가드 필수
```

**해결**:
```markdown
## 개발 가이드라인

### 우선순위

**🔴 필수 (MUST)**:
- 보안 취약점 없음
- 타입 안정성 (TypeScript strict 모드)
- 모든 public API에 테스트 존재

**🟡 권장 (SHOULD)**:
- 함수는 50줄 이하 (복잡한 로직은 예외)
- 외부 라이브러리는 검증된 것만 (npm trends 확인)
- 중요 로직에 주석

**🟢 선택 (MAY)**:
- JSDoc 주석 (public API는 권장)
- 성능 최적화 (병목이 확인된 경우)

**균형 잡기**:
코드 품질과 개발 속도의 균형을 유지하세요.
너무 엄격한 규칙은 생산성을 저하시킬 수 있습니다.
```
